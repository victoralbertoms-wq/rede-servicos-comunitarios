🔒 Regras de Segurança

⚠️ Nunca usar get() aninhado dentro das rules — causa falhas silenciosas. As regras usam campos diretos do documento (resource.data.adminIds, resource.data.userId). 
// Comunidades — editável por adminIds
allow update: if request.auth.uid in resource.data.adminIds;

// Serviços / Empresas — editável pelo dono
allow update: if resource.data.userId == request.auth.uid;

// Leitura — qualquer usuário autenticado
allow read: if request.auth != null;

Para aplicar as regras: cd firebase && firebase deploy --only firestore:rules

🧠 Decisões de Arquitetura

🚫 Sem índice composto para where + orderBy

Quando há filtro where('communityId', '==', id), o orderBy é omitido para evitar exigência de índice composto. A ordenação é feita client-side com .sort(). Ver getServices() e getCompanies() em firestoreService.js.

🔍 Busca 100% client-side com useMemo

Carrega todos os documentos uma vez e filtra localmente com useMemo. Não faz chamadas Firestore por tecla digitada. Evita leituras desnecessárias e problemas de índice.

☁️ Cloudinary em vez de Firebase Storage

Firebase Storage exige plano pago (Blaze). Cloudinary tem plano gratuito generoso. O upload usa preset Unsigned — sem backend necessário. Configurar em: Cloudinary Console → Settings → Upload → Upload presets → rede-servicos → Signing Mode = Unsigned.

✅ Status de aprovação

Serviços e empresas criados com status: 'pending'. Admin aprova pelo Painel Admin (/admin). Somente status: 'approved' aparece publicamente. O dono sempre vê o próprio item.

🏘️ adminIds no documento de comunidade

Ao criar comunidade, adminIds: [user.uid] é salvo no documento. As regras Firestore e o frontend usam esse campo para checar permissão de edição/exclusão sem chamadas extras.

⚙️ CI/CD — GitHub Actions

Arquivo: .github/workflows/deploy.yml

Fluxo automático a cada push na branch main:

git push → GitHub Actions →
  1. npm ci (instala dependências)
  2. Cria .env a partir dos Secrets do GitHub
  3. npm run build (gera web/dist/)
  4. cp web/dist → firebase/dist/
  5. firebase deploy --only hosting

Secrets necessários no GitHub

Em: github.com/[repo]/settings/secrets/actions

FIREBASE_API_KEY · FIREBASE_AUTH_DOMAIN · FIREBASE_PROJECT_ID · FIREBASE_MESSAGING_SENDER_ID · FIREBASE_APP_ID · CLOUDINARY_CLOUD_NAME · CLOUDINARY_UPLOAD_PRESET · FIREBASE_TOKEN · FIREBASE_SERVICE_ACCOUNT

🛡️ Como Tornar Usuário Admin

  1. Acesse Firebase Console → Firestore Database → coleção users
  2. Abra o documento do usuário desejado
  3. Edite o campo role para "admin"
  4. Salve
✅ O usuário terá acesso ao Painel Admin em /admin e poderá aprovar cadastros, gerenciar usuários e criar comunidades.
